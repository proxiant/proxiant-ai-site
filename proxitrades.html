How it's wired.

Five layers, each replaceable. The kill switch always works even when the rest is broken.

SIGNAL SOURCES

Three signal modes per strategy

A TOML field decides where the signal comes from — without touching strategy code.

  • tv — TradingView Pine alert hits the webhook
  • native — In-process Python on the websocket bar feed
  • either — Both; idempotent client_order_id collapses duplicates
RISK STACK

Three-layer kill

Defense in depth. Operator action, automated dead-man, broker-side bracket.

  • proxi kill — Operator CLI, imports zero strategy code
  • Dead-man — Flatten everything if no heartbeat in N seconds
  • Protective bracket — Stop+TP posted alongside every entry
ADAPTIVE LOOP

Hypothesis → Panel → Factory → Decay

The system mints, validates, and retires strategies faster than the market arbitrages them away.

  • Claude proposes setups; deterministic factory walks the param grid
  • 3-persona panel + adversarial Bull/Bear/Referee debate gates each card
  • Live monitor auto-disables on Sharpe drift > threshold
DAILY LEARNING

Reports that catalogue what works

Every day at 20:00 UTC, the system writes a structured report and emails it.

  • Per-strategy verdict: WORKING / NOT_WORKING / MIXED, with WHY
  • LLM-suggested config diffs as machine-applicable change bundles
  • Reflection memory store retrieves prior lessons by regime similarity

What this is not.

ProxiTrades is not a get-rich-quick system. The math reality:

  • "1% per day on capital" implies ~1,170% annualized — about 18× the best documented hedge fund track record. It is not a viable target.
  • Plausible defensible targets are Sharpe 1.0–2.0 with controlled drawdown. V7 lands at OOS Sharpe 2.06 with 6% DD on a specific universe over a specific window. Out-of-sample variance is real and the live monitor is the safety net.
  • Live trading requires two operator switches: env = "live" in the strategy TOML AND LIVE_TRADING=true in the environment. The apply engine refuses any change that would set both at once.
